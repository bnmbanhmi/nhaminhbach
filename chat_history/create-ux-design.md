   * BottomSheet (Drawer for Mobile Details)
   * Toast (for Success/Error messages)
   * Dropdown (for basic menus)

  2. Custom "NhaMinhBach" Components (The Special Sauce):
  These are the components that deliver the unique UX:

   * `OmniSearchInput` (The Crown Jewel):
       * Purpose: Handles text entry, regex parsing, and suggestion display.
       * Complex Logic: Needs a "Parser Hook" to detect State 1 (Empty), State 2 (Typing/Regex), State 3 (Results).
       * Visuals: Morphs from large Hero Input (Home) to Compact Navbar Input (Results).

   * `ListingCard` (The Workhorse):
       * Purpose: Displays property info in the feed.
       * Complex Logic: Internal image swiper (swiper.js), "Like" state, and "Zalo Contact" deep link.
       * Visuals: High-density grid layout, clear price badge.

   * `SmartChipGroup` (The Guide):
       * Purpose: Shows suggested filters or intent disambiguation.
       * Visuals: Horizontal scroll container with "Pill" shaped buttons.

   * `QCSideBySide` (For Admin):
       * Purpose: Admin verification interface.
       * Visuals: Split-screen layout (Raw HTML frame vs. Parsed Form).

  3. Implementation Roadmap:
   * Phase 1 (The Skeleton): Build OmniSearchInput and ListingCard. These are 80% of the user experience.
   * Phase 2 (The Flow): Build BottomSheet detail view and SmartChipGroup for filtering.
   * Phase 3 (The Admin): Build QCSideBySide for the verified workflow.

✦ I will now define the UX Consistency Patterns, ensuring they align with the "Hybrid Social-Concierge" direction and the "Organized Density" philosophy.

  UX Pattern Analysis:

  1. Button Hierarchy:
   * Primary (Coral): Used sparingly for "Go" actions (Contact, Verify, Search).
   * Secondary (Outlined/Gray): Used for navigation or secondary filters.
   * Ghost (Text only): Used for "See more" or "Cancel."
   * Floating Action Button (FAB): The "Quick Jump" button that appears when a GeoID is detected.

  2. Feedback Patterns (The "Her" Voice):
   * Success: Subtle "Toast" at the bottom center (e.g., "Copied to clipboard").
   * Error: Inline validation in the Omnibox (e.g., "Address not found").
   * Empty State: Never show a blank screen. Always show "Smart Suggestions" or "Popular Searches."

  3. Form Patterns (The Concierge):
   * Input: Single-line inputs with floating labels are too formal. We use "Chips" inside the Omnibox for filters.
   * Validation: Real-time. As the user types "30", suggest "30 million?" or "30m2?".

  4. Navigation Patterns (Mobile-First):
   * Bottom Nav: 3 tabs only (Home/Search, Favorites, Admin/Profile).
   * Transition: Slide-left for deeper navigation (Details), Fade for lateral navigation (Tabs).
   * Back: Swipe-to-back gesture support is mandatory.

  5. Modal/Overlay Patterns:
   * Bottom Sheet: The default for everything on mobile (Filters, Details, Contact). Keeps context.
   * Center Modal: Only for desktop "Quick View" or critical confirmations (Delete).

  Responsive & Accessibility Analysis:

  1. Responsive Strategy (Mobile-First):
   * Philosophy: The mobile experience is the primary experience. Desktop is an enhancement, not the default.
   * Mobile (<768px): Single-column feed. Navigation via bottom bar. Search bar collapses to a "Search Icon" or floating header on scroll.
   * Tablet (768px - 1024px): Two-column masonry grid. Sidebar navigation emerges.
   * Desktop (>1024px): Three/Four-column masonry grid. The "Omnibox" becomes a persistent, centered hero element. "Side-by-Side" view active for Admin tools.

  2. Breakpoint Strategy (Tailwind Default):
  We will stick to standard Tailwind breakpoints to ensure developer velocity and predictability:
   * sm: 640px (Large phones)
   * md: 768px (Tablets)
   * lg: 1024px (Laptops)
   * xl: 1280px (Desktops)

  3. Accessibility Strategy (WCAG AA):
   * Color Contrast: Our "Future Warmth" palette (Charcoal on Cream) naturally meets AA standards (4.5:1). Coral text must only be used on White/Cream if
     bold/large, otherwise it serves as a background for White text.
   * Keyboard Nav: The "Omnibox" must be 100% drivable via keyboard (Arrows to select suggestion, Enter to jump).
   * Screen Readers: All "Icon-only" buttons (like the Zalo icon) must have aria-label="Contact via Zalo".

  4. Testing Strategy:
   * Touch Testing: Verify that "Swipe to Close" on bottom sheets works on actual devices, not just simulators.
   * Thumb Zone: Ensure the "Quick Jump" and "Contact" buttons sit within the natural reach of a thumb on a 6.7" screen.
